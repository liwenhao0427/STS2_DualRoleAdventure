# LocalMultiControl 项目结构与实现流程说明（当前版本）

本文档用于记录当前代码结构、关键功能和主流程，方便后续迭代时快速定位风险点。

## 1. 项目目标

- 目标：将原本“联机双人”的控制链路在单进程内本地复用，实现双角色切换控制。
- 原则：优先保证输入路由与状态同步安全，再逐步增加玩法层同步。
- 当前策略：
  - 战斗内允许在两名角色之间切换控制；
  - 事件流默认收束为单事件视角（以1号位为主）；
  - 关键多人同步链路在本地回环模式下按需裁剪，避免单进程生命周期冲突。

## 2. 主要目录与职责

- `Scripts/Entry.cs`
  - 模组入口，负责 Harmony 补丁注册。
- `Scripts/Runtime/`
  - 运行态上下文与会话控制核心。
  - 重点：`LocalMultiControlRuntime.cs`、`LocalSelfCoopContext.cs`、`LocalLoopbackHostGameService.cs`。
- `Scripts/Patch/`
  - Harmony 补丁集合，按系统分散到具体文件。
  - 包含大厅、战斗、事件、奖励、休息区、药水、遗物、输入等。

## 3. 关键运行态对象

### 3.1 LocalSelfCoopContext

- 职责：本地双人总开关、主副角色ID、读档/开局标记、单事件流配置。
- 作用：所有补丁是否生效，均以该上下文作为第一层条件判断。

### 3.2 LocalMultiSessionState

- 职责：维护“当前操控角色”与可切换角色顺序。
- 约束：当前按双人模型（ExpectedPlayerCount=2）进行初始化与切换。

### 3.3 LocalMultiControlRuntime

- 职责：切换控制上下文、刷新 UI、同步各类 Synchronizer 的本地玩家ID。
- 关键点：
  - 切换入口：`SwitchNextControlledPlayer` / `SwitchPreviousControlledPlayer` / `SwitchControlledPlayerTo`。
  - 核心落地：`ApplyControlContext`。
  - 战斗保护：`CanSwitchDuringCombat`、`TrySwitchCombatPlayer`、`RefreshCombatUiForControlledPlayer`。

## 4. 主流程说明

### 4.1 从大厅到开局

1. 在联机菜单进入“与自己联机”流程。
2. 创建本地回环网络服务（host/client 逻辑在同进程模拟）。
3. 初始化双人队伍与角色选择。
4. BeginRun 时接管为本地双人运行态，并初始化会话。

### 4.2 运行中切换控制

1. 监听热键（`[` / `]` / `/`）触发切换。
2. 若在战斗中：
   - 仅允许在当前 `CombatState` 的两个玩家之间切换；
   - 仅在 `PlayPhase` 且无拖牌/选牌/目标选择时切换。
3. 进入 `ApplyControlContext`：
   - 更新 `LocalContext.NetId` 与 loopback sender；
   - 同步各 Synchronizer `_localPlayerId`；
   - 刷新战斗UI、顶部栏、事件房间、商店上下文。

### 4.3 事件处理（单事件流）

- 目标：事件界面固定以主角色为基准，避免两套事件并行导致流程错位。
- 补充策略：非战斗选牌（附魔/升级/改造等）强制本地处理，避免 `Awaiting remote choice` 卡死。

### 4.4 休息区处理

- 当前策略：
  - 去除 `Mend` 这类“给另一角色单独操作”的分叉；
  - `Heal` 维持“单次选择，全体恢复”。

### 4.5 战斗处理

- 自动回合切换：一个角色结束回合后自动切到下一角色；双人都结束后等待敌方回合。
- UI重绑：切人时重建当前手牌与能量显示并重算结束回合按钮状态。
- 稳定性防线：
  - 切换时机保护（禁止在拖牌/选牌中切换）；
  - UI刷新失败时回滚上下文，避免 owner 与显示错绑。

## 5. 当前已知高风险点（后续修改需谨慎）

1. 战斗中切换与 `NMouseCardPlay` 生命周期耦合强。
   - 在拖牌、目标选择、手牌选择期间切换会导致对象销毁竞争。
2. 多人意图与远端光标链路（`NMultiplayerPlayerIntentHandler` / `NRemoteMouseCursorContainer`）
   - 在本地双人场景下易触发 `ObjectDisposedException`，已采取禁用策略。
3. 事件中的玩家选择同步
   - 单事件流下若回到远端等待分支，极易卡在 `Awaiting remote choice`。
4. 读档路径
   - 观察到 `SetUpSavedMultiPlayer` 的重复触发风险（`State is already set`），需后续继续收敛。
