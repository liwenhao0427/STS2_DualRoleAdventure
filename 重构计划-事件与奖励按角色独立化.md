# 重构计划：事件/休息/战利品按角色独立化（仅规划）

更新时间：2026-03-18
分支：`codex/refactor-role-independent-flow`
状态：规划中（本文件不包含代码实现）

## 1. 本次重构目标

在现有本地多控基础上，完整接入原多人联机的“多角色独立决策”语义，重点覆盖事件、休息房间与战斗胜利后战利品阶段，避免“一人选择全员同步”。

## 2. 需求拆解（来自本次会话）

1. 事件选择默认改为“每个角色独立选择”，流程为“当前角色确认后切到下一角色”。
2. 在事件触发时新增“同步到全部”开关（默认关闭）；开启时才沿用旧的全员同步策略。
3. 休息房间（Rest Site）改为每个角色独立选择，保留原多人联机选项集合。
4. 战斗胜利后的战利品改为每个角色独立获取（金币、卡牌、药水均不同步）。
5. 金币、遗物不再复制；药水不再仅给角色1，且不再额外扩展 2 个药水槽位。
6. 宝箱房间保留“5人以上特判”，仍复制角色1选项（维持现状）。

## 3. 设计约束与兼容策略

- 原则：优先复用多人联机已有同步器与命令链路，避免重写整段业务流程。
- 默认行为切换：将“事件自动同步到全部”从默认开启调整为默认关闭（仅在开关开启时生效）。
- 兼容旧存档：
  - 新增的事件同步开关应提供缺省值（`false`）。
  - 对已存在流程做幂等保护，重复触发不应导致角色顺序错乱。
- 非目标：
  - 本轮不扩展 Daily / Custom。
  - 本轮不新增 UI 大改，仅补最小必要开关与状态提示。

## 4. 实施阶段（计划）

## Phase A：事件链路独立化

- 目标：事件从“全员同步选择”改为“按角色顺序独立选择”。
- 关键点：
  - 增加事件上下文中的“同步到全部”开关（默认关闭）。
  - 角色 A 完成事件后，自动切换到角色 B（直到所有本地角色完成）。
  - 开关开启时，走旧同步逻辑并保持兼容。
- 验收：
  - 同一事件内可看到角色逐个完成；
  - 默认不会一人操作导致全员同步；
  - 开关开启后可恢复旧行为。

## Phase B：休息房间独立化

- 目标：每个角色单独进行 Rest Site 选择。
- 关键点：
  - 保留原多人联机选项，不裁剪。
  - 每个角色进入自己的结算分支，避免共享状态覆盖。
- 验收：
  - 休息区选择可按角色依次完成；
  - 不再出现“仅一人可操作/结算”。

## Phase C：战利品独立化

- 目标：战斗胜利后金币/卡牌/药水按角色独立发放与选择。
- 关键点：
  - 取消当前复制/镜像分发策略。
  - 药水归属按当前角色，不再仅给角色1，不再扩容额外槽位。
  - 金币与遗物不再复制。
- 验收：
  - 每个角色独立完成奖励领取；
  - 资源归属与当前角色一致；
  - 无重复发放或漏发。

## Phase D：宝箱特判回归验证

- 目标：保留“5人以上复制角色1选项”的既有特判。
- 验收：
  - 5人以上宝箱逻辑行为与改动前一致；
  - 不受 A/B/C 阶段改动回归影响。

## 5. 风险与回滚预案

- 风险 1：事件链路切人时机错误，导致奖励弹窗与当前控制角色错位。
  - 预案：统一在“事件流程完成且阻塞弹窗关闭后”再执行切人。
- 风险 2：奖励独立化后与既有镜像补丁冲突。
  - 预案：按模块分批下线镜像逻辑，并保留开关快速回退。
- 风险 3：Rest Site 与 Event 共享上下文状态，可能出现重复提交。
  - 预案：所有接管/释放控制权接口保持幂等，并增加关键日志断言。

## 7. 交付节奏

1. 先完成 Phase A，验证事件链路稳定后再进入 Phase B。
2. Phase B 与 Phase C 分两次提交，降低回归面。
3. 最后执行 Phase D 回归与全链路冒烟验证。
