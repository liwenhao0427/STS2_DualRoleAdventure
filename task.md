# LocalMultiControl 开发任务单

## 目标（阶段 A）

- 在联机入口新增“与自己联机”。
- 首版固定 2 人本地多控（标准局）。
- 战斗内支持 `[` / `]` 切换当前操控角色。
- 奖励/商店/事件等流程尽量复用多人同步器，资源归属到对应角色。

## 已确认约束

- 首版人数固定 2 人。
- 允许重复角色。
- 共享事件采用自动代投。
- 暂不优先支持 Daily / Custom。

## 任务拆解

### T1 - 本地多控会话骨架

- [x] 新增本地会话状态管理（2 人固定槽位）。
- [x] 维护当前操控角色 `CurrentControlledPlayerId`。
- [x] 提供切换接口：`SwitchNextPlayer()` / `SwitchPrevPlayer()`。

### T2 - 联机入口注入

- [x] 在联机 Host 菜单注入“与自己联机”按钮。
- [x] 按钮进入标准局角色选择流程。
- [x] 默认创建 2 人本地队伍（iter-03 接入本地回环后完成）。

### T3 - 本地回环网络适配

- [x] 提供 `INetGameService` 的本地回环实现（仅本 Mod 使用）。
- [x] 支持按当前操控角色发送消息（可指定 senderId）。
- [x] 消息处理保持幂等，避免重复分发。

### T4 - 控制上下文切换

- [x] 切换角色时同步更新 `LocalContext.NetId`。
- [x] 确保手牌、结束回合、目标选择跟随当前角色（当前为最小闭环，持续补充）。
- [ ] UI 增加当前操控角色提示。

### T5 - 非战斗流程归属验证

- [ ] 奖励选择归属正确。
- [ ] 商店购买/删牌归属正确。
- [ ] 事件选择（含共享事件自动代投）归属正确。

### T6 - 日志与自检

- [ ] 关键路径日志前缀统一 `[LocalMultiControl]`。
- [ ] 本地构建通过：`dotnet build LocalMultiControl.csproj -c Debug`。
- [ ] 格式检查通过：`dotnet format LocalMultiControl.csproj --verify-no-changes`。

## 开发顺序

1. 先做 T1 + T2，打通入口到开局路径。
2. 再做 T3 + T4，打通战斗内切换闭环。
3. 最后做 T5 + T6，补齐流程与验证。

## 迭代与验收规则（新增）

- 每一步都必须产出一个可在游戏中直接验证的小版本（可运行、可操作、可观察日志）。
- 每一步完成后都执行一次完整最小验证：
  - 能进游戏并加载 Mod。
  - 触发该步对应功能路径。
  - 在 `godot.log` 看到对应 `[LocalMultiControl]` 关键日志。
- 每一步都单独提交一次 commit，提交信息需标注迭代编号（如 `iter-01`、`iter-02`）。
- 若某步无法形成可验证闭环，必须先补“临时可验证开关/入口/日志”再进入下一步。
